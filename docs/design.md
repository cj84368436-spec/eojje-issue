# 어제 이슈 - 리빌딩 설계 문서

2026-06-11 리빌딩 기준. 기존 구현(`문서\앱토스`)은 참고만 하고 전면 재작성했다.

## 1. 리빌딩 방향

기존 구현의 실측 문제 (2026-06-11자 기존 출력 기준):

| 문제 | 기존 | 리빌딩 |
|---|---|---|
| 3개월 지난 기사 혼입 | 최신성은 점수로만 반영 | 수집 단계 40시간 하드 컷 |
| 사진 캡션이 요약으로 노출 | 캡션 감지 없음 | `사진은/모습` 패턴 감지 후 요약 제외 |
| 제목이 "안전 방"처럼 중간 절단 | 36자 강제 절단 | 제목은 줄이지 않음, API가 자른 제목은 기사 자체를 버림 |
| 출처가 전부 "네이버 뉴스 검색" | 출처 미해석 | originallink 도메인 → 실제 언론사명 + 신뢰 등급 |
| 주목도 전부 99 쏠림 | 절대 점수 클램프 | 풀 내 백분위 스케일링 (42~97) |
| 계엄/특검 기사 민감도 "낮음" | 민감어 8개 | 높음/중간 2단계 민감어 + 헤드라인에서 높음 제외 |
| 품질검사 "문제 없음"인데 실제 문제 있음 | 검사 항목 협소 | 발행 차단(block)/주의(warn) 2단계, 차단 시 CI 실패 |
| 매일 갱신 불가 (빌드 포함 JSON만) | - | 원격 URL 우선 로드 + GitHub Actions 일일 발행 |

```text
│   ├─ 1위: 리드 카드 (제목+요약+메타)
│   └─ 나머지 대표 이슈: 순위 행
├─ 정치 / 경제 / 사회 / 문화 / 연예: 각 6개 이슈 카드
├─ AI: 업데이트 소식이 있을 때 6개 이슈 카드, 없으면 비어 있어도 정상
├─ 저장: 저장한 이슈 (localStorage 스냅샷, 날짜 무관 유지)
└─ 상세 바텀시트: 30초 브리핑·핵심 맥락·키워드·원본 링크·저장
```

상태: 로딩(스켈레톤) / 오류(재시도 버튼) / 갱신 지연 배너(데이터 2일 이상 경과) / 빈 저장함.

## 4. 파이프라인 단계 (pipeline/src)

1. `collect.js` - 네이버 뉴스 API. 카테고리×쿼리 5개×(date+sim 정렬). 40시간 컷, 잘린 제목 폐기, 도메인→언론사 해석
2. `dedupe.js` - URL/정규화 제목/토큰 유사도 중복 제거. 같은 사건 보도 수는 clusterSize로 보존(화제성 신호)
3. `classify.js` - 키워드 가점 - 타 카테고리 감점, fit 점수 보존
4. `summarize.js` - 추출형 요약: 완결 문장만. 실패 시 빈 요약(후순위). 키워드 3개
5. `score.js` - heat = 반복보도+최신성+대화소재+적합성+출처등급-홍보성감점 → 백분위 42~97. 민감도 산정
6. `select.js` - 활성 카테고리당 6개(요약 우선, 사건 중복 금지), 헤드라인은 활성 카테고리 대표 1개씩(완결 요약 필수, 높음 제외, 교차 중복 금지)
7. `report.js` - 품질 지표/경고 산출, block 경고 시 exit 1
8. `publish.js` - output + history 아카이브 + app/public 복사

## 5. 운영

- GitHub Actions cron 06:30/08:00 KST → 파이프라인 → 검증 → GitHub Pages 배포
- 앱은 `REMOTE_DATA_URL` 우선, 실패 시 번들 fallback, 2일 이상 경과 시 배너
- 품질 리포트: `output/quality-report.md`(사람용) + `.json`(지표 추적용)
- AI 요약: 사용 안 함 (추출형만). 추후 붙일 경우 비용·심사 검토 후 별도 모듈로
